# AI 아키텍처

이 문서는 On My Way AI 요청의 출시 목표 구조와 안전 계약을 설명합니다. 문서에 API나 상태 전이를 정의했다고 해서 배포가 완료된 것은 아닙니다. 실제 완료 여부는 Worker 라우트, 저장소 구현, 자동 테스트와 Preview 응답으로 확인합니다.

## 1. 구성과 신뢰 경계

```text
브라우저 UI
  → Cloudflare Worker 인증·출처 검사
  → 서버 action 매핑
  → plan·trial·AI 크레딧 판정 및 예약
  → OpenAI API
  → 응답 구조 검증
  → 크레딧 확정 또는 예약 해제
  → 최신 usage를 포함한 응답
```

브라우저는 화면을 표시하고 `requestId`와 기능 입력을 보냅니다. 다음 값은 서버만 결정합니다.

- 인증된 사용자 ID와 역할
- 현재 plan과 유효한 trial·subscription 상태
- 요청 라우트에 해당하는 action
- action별 크레딧 비용
- 일일·월간·체험 잔액
- 차감 확정, 해제와 초기화 시각

클라이언트가 보낸 `plan`, `price`, `creditCost`, `remaining` 또는 사용자 ID는 권한 판정에 사용하지 않습니다.

## 2. 중앙 정책

모든 AI 라우트, 사용량 조회, UI 표시, 관리자 조회와 테스트는 `plan-policy.mjs`의 `PLAN_CONFIG`, `AI_CREDIT_COSTS`, `AI_ACTION_LABELS`, `CREDIT_POLICY_VERSION`, `DEFAULT_TIME_ZONE`을 기준으로 해야 합니다. `ai-credits-service.mjs`가 사용자 상태 정규화, 한도 판정, 예약·확정·해제를 담당하며 action별 숫자를 다시 선언하지 않습니다.

| action | 비용 |
| --- | ---: |
| `companion_chat` | 1 |
| `create_daily_step` | 2 |
| `revise_plan` | 2 |
| `recovery_plan` | 3 |
| `create_plan` | 4 |
| `reschedule_plan` | 4 |

Free, Pro와 Trial의 한도 및 초기화 규칙은 [가격 및 AI 크레딧 정책](pricing-and-credits.md)을 따릅니다. action 별칭이 필요한 기존 라우트는 중앙 매핑에서만 변환하며 각 핸들러에 숫자를 중복 하드코딩하지 않습니다.

## 3. 요청 계약

### 사용량 조회

`GET /api/ai/usage`는 인증된 사용자의 현재 plan, 체험 자격·상태, 일일·월간 사용량, 초기화 시각과 action 비용을 서버 계산값으로 반환하는 목표 API입니다. 구현·배포 여부는 라우트 테스트와 Preview에서 확인해야 합니다.

### AI 실행

각 AI 쓰기 요청은 다음을 포함합니다.

- 클라이언트가 생성한 충분히 고유한 `requestId`
- 해당 기능에 필요한 입력
- 선택적으로 화면에서 사용할 문맥

서버 응답은 기능 결과 외에 다음을 포함해야 합니다.

- 실제 `chargedCredits`
- 최신 일일·월간 또는 체험 잔여량
- 동일 `requestId` 재요청을 구분하는 명시적 오류 코드

클라이언트 `requestId`는 필수이며, 누락되거나 허용 형식이 아니면 공급자를 호출하기 전에 거부합니다.

## 4. 예약 상태 머신

```text
received
  ├─ unauthenticated / forbidden → rejected, 차감 없음
  ├─ insufficient / limit reached → rejected, 공급자 호출 없음
  ├─ duplicate in progress → 409 AI_REQUEST_IN_PROGRESS, 재차감 없음
  ├─ duplicate completed → 409 AI_REQUEST_ALREADY_COMMITTED, 재차감 없음
  ├─ duplicate released → 409 AI_REQUEST_PREVIOUSLY_RELEASED, 재차감 없음
  └─ reserved
       ├─ valid AI response → committed
       └─ timeout / provider error / invalid response / server error → released
```

예약에는 사용자 ID, `requestId`, action, 비용, 생성·만료 시각과 상태를 기록합니다. 성공은 단순 HTTP 2xx가 아니라 기대한 응답 구조 검증까지 통과한 경우만 의미합니다.

모델 호출 실패, timeout, JSON 파싱 실패, 스키마 불일치와 Worker 예외는 확정 차감하지 않습니다. 이미 예약했다면 같은 서버 흐름에서 해제하고, 연결 중단으로 10분 넘게 남은 예약은 다음 원장 접근 때 정리합니다.

## 5. 멱등성과 재시도

- 멱등성 범위는 사용자 ID와 `requestId`의 조합입니다.
- 같은 조합은 action이나 비용을 바꿔 재사용할 수 없습니다.
- 완료된 요청 재전송은 공급자 API를 다시 호출하거나 다시 차감하지 않습니다.
- 진행 중 요청 재전송은 새 예약을 만들지 않습니다.
- 해제된 요청은 같은 ID로 다시 실행하지 않고 새 ID를 요구합니다.
- 원래 AI 결과 전문은 멱등성 원장에 저장하지 않으므로 완료된 요청을 재전송해도 결과를 재생하지 않고 명시적 409를 반환합니다.
- 10분 안전 창이 지나 자동 해제된 예약의 늦은 결과는 확정하지 않습니다.
- 완료 ID는 최소한 사용자에게 재시도가 가능한 기간 동안 보존합니다.

이 규칙은 사용자의 연속 클릭과 같은 ID 재전송의 중복 호출·차감을 막습니다. 다만 성공 응답이 유실된 경우 원래 결과를 재생하지 않으므로 클라이언트는 409를 새 성공으로 오인해서는 안 됩니다. 서로 다른 ID의 동시 요청을 전역적으로 원자 직렬화하지도 않습니다.

## 6. 저장소와 마이그레이션

계정, 체험, 구독과 AI 사용량은 `USERS_KV`의 서버 사용자 ID에 연결합니다. 별도 사용량 KV key를 추가하지 않고 기존 `user:<id>` JSON 내부의 `aiCredits`를 사용합니다. 이 필드는 `policyVersion`, `timeZone`, `trial`, `usage.day`, `usage.month`, `requests`, `metrics.byAction`, `metrics.tokens`, `metrics.calls`의 논리 구조를 가집니다. 브라우저 localStorage는 캐시나 표시 보조 수단일 뿐 권한 원장이 아닙니다.

기존 데이터는 schema-on-read로 정규화합니다.

- 기존 Pro 보존
- 아직 유효한 Trial은 원래 종료 시각까지 보존한 뒤 Free 전환
- 만료·불완전 Trial과 일반 기존 사용자는 Free
- 레거시 브라우저 에너지를 서버 크레딧으로 변환하지 않음
- 새 서버 일·월 기간을 시작하고 plan별 할당을 한 번만 지급
- 정책 버전과 기간 키로 반복 지급 방지
- 목표, 계획, 기록과 구독 증빙 보존

상세 전환·롤백 규칙은 [가격 및 AI 크레딧 정책](pricing-and-credits.md#6-kv-schema-on-read-마이그레이션)을 따릅니다.

## 7. 시간

서버는 사용자 레코드의 유효한 IANA `timezone`을 우선하고, 값이 없거나 잘못되었으면 `Asia/Seoul`을 사용합니다. 결제가 비활성인 현재 단계에서는 해당 시간대의 달력 월을 크레딧 기간으로 사용합니다.

- 일일 경계: 사용자 시간대의 00:00
- 월간 경계: 사용자 시간대의 매월 1일 00:00
- Trial: 시작 즉시부터 정확히 24시간

테스트는 시스템 시계를 직접 기다리지 않고 주입 가능한 clock을 사용합니다. 결제가 활성화되면 Pro 월간 기간을 결제 주기로 전환할 수 있으므로 날짜 계산은 AI 핸들러와 분리합니다.

## 8. KV 동시성 제한

같은 Worker isolate에서는 사용자별 in-process lock으로 크레딧 작업을 직렬화합니다. 하지만 Cloudflare KV는 강한 트랜잭션 원장이 아니므로 서로 다른 isolate·리전에서 사용자 상태를 읽고 예약을 추가한 뒤 다시 쓰는 동안 각 요청이 같은 이전 잔액을 볼 수 있습니다. 다음 사항은 구현 후에도 남을 수 있는 위험입니다.

- 서로 다른 `requestId`의 동시 승인으로 일시적인 초과 사용
- 마지막 쓰기가 앞선 사용량을 덮어쓰는 갱신 유실
- 지역 간 전파 지연으로 잠시 오래된 잔액 표시
- 예약 만료와 실제 공급자 완료의 경합

요청별 멱등성, 보수적인 잔액 검사, 짧은 예약 만료와 모니터링으로 빈도를 줄일 수 있지만 원자성을 보장하지는 않습니다. 결제 활성화 또는 트래픽 증가 전 사용자별 Durable Object나 D1 트랜잭션 ledger로 이동하는 것이 권장되는 장기 해결책입니다.

## 9. 오류와 사용자 경험

| 상태 | 서버 동작 | UI 원칙 |
| --- | --- | --- |
| 비로그인 | 401, 차감 없음 | 로그인 안내 |
| 일일 한도 도달 | 공급자 호출 없이 거부 | 다음 일일 초기화 시각 표시 |
| 월간/체험 크레딧 소진 | 공급자 호출 없이 거부 | 현재 잔액과 선택 가능한 plan 안내 |
| 체험 만료 | Free로 정규화 | Free 한도와 Pro 상태 안내 |
| 공급자 timeout·오류 | 예약 해제 | 재시도 가능, 차감되지 않았음을 표시 |
| 응답 구조 오류 | 예약 해제 | 일반 오류 처리, 잘못된 결과 미적용 |
| 저장소 오류 | 보수적으로 거부 | 결제를 유도하지 않고 잠시 후 재시도 안내 |

AI가 계획 변경안을 만들더라도 사용자가 승인하기 전 기존 계획을 자동 덮어쓰지 않습니다.

## 10. 개인정보와 분석

- OpenAI에는 기능 수행에 필요한 최소 문맥만 보냅니다.
- API 키와 내부 사용자 상태를 브라우저에 노출하지 않습니다.
- 로그에 OAuth token, 세션, API 키, 전화번호, 이메일이나 전체 프롬프트를 기록하지 않습니다.
- 분석 이벤트에는 action, 결과 상태, 익명화한 정책 버전과 차감량 정도만 포함합니다.
- 목표 원문, AI 대화 원문과 사용자 자유 입력은 분석 이벤트로 보내지 않습니다.

## 11. 관측성과 관리자 화면

운영에서 필요한 최소 지표는 다음과 같습니다.

- action별 요청·성공·실패·timeout 수
- 예약, 확정, 해제와 만료 수
- 중복 `requestId` 차단 수
- plan별 일일·월간 크레딧 사용 분포
- 한도 도달 수
- 추정 token 사용량과 비용
- 음수 잔액, 미해제 예약과 비정상 초과 사용
- 적용 중인 정책 버전

관리자 화면의 샘플 KPI를 실제 값으로 오해하지 않습니다. 실제 사용량 조회가 연결되기 전에는 샘플임을 유지합니다.

## 12. 검증과 롤백

필수 자동 테스트:

- 중앙 plan/action 설정 exact 값
- 인증·출처 검사 후에만 사용량 접근
- Free 5/2, Pro 250/30, Trial 15/24시간
- 각 action 1/2/2/3/4/4 차감
- 부족 상태에서 공급자 호출 0회
- 실패와 잘못된 응답에서 예약 해제
- 동일 `requestId` 중복 차감·호출 없음
- 클라이언트 plan·비용 조작 무시
- 사용자 시간대(기본 `Asia/Seoul`)의 일·월 경계와 체험 만료
- schema-on-read 반복 실행의 멱등성
- 기존 Pro와 유효 Trial 보존

문제가 발견되면 AI 쓰기를 중단하고 직전 Worker 버전으로 되돌립니다. 새 사용량·예약 데이터와 이전 필드는 즉시 삭제하지 않으며, 진행 중 예약을 정리하고 공급자 호출 기록과 대조합니다. 새 크레딧을 레거시 localStorage 에너지로 역변환하지 않습니다.
