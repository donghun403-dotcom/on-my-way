# 가격 및 AI 크레딧 정책

이 문서는 On My Way의 Free, Pro, 24시간 Pro 체험과 AI 크레딧에 대한 출시 계약입니다. 정책을 설명하는 문서일 뿐이며, 특정 UI·서버 기능이 배포 완료되었다는 증거로 사용하지 않습니다. 출시 판정은 코드, 자동 테스트, Preview와 운영 설정을 함께 확인해야 합니다.

## 1. 단일 정책

### Free

- 가격: 무료
- 최초 가입 보너스: AI 크레딧 5개
- 이후 매월 제공: AI 크레딧 5개
- 일일 사용 한도: AI 크레딧 2개
- 목표: 최대 1개
- 활성 계획: 최대 1개
- 기본 기록 기능 제공
- 전체 일정 재조정, 회복 계획, 상세 통계, Pro 전용 올리 개인화는 제한
- 미사용 월간 크레딧은 다음 달로 이월하지 않음

가입 보너스 5개와 첫 달 월간 5개를 중복 지급하지 않습니다. 신규 Free 사용자의 첫 크레딧 기간에 5개를 한 번 제공하고, 다음 달부터 월 5개를 제공합니다.

### Pro

- 가격: 월 4,900원
- 매월 AI 크레딧 250개
- 일일 사용 한도: AI 크레딧 30개
- 목표와 활성 계획 수 제한 없음
- 전체 일정 재조정, 회복 계획, 상세 기록·통계와 올리 개인화 제공
- 미사용 월간 크레딧은 다음 달로 이월하지 않음
- AI 사용에는 월간·일일 크레딧 한도가 적용됨

### 24시간 Pro 체험

- 결정론적 서버 사용자 ID 기준 계정당 1회만 시작 가능
- 시작 시점부터 정확히 24시간
- 체험 전용 AI 크레딧 총 15개
- 일일 상한은 30개지만 체험 총량 15개를 넘을 수 없음
- 24시간이 지나거나 15개를 모두 사용하면 종료
- 종료 후 활성 유료 구독이 없으면 Free로 전환
- 남은 체험 크레딧은 종료 즉시 소멸
- 결제수단을 등록하지 않은 체험은 자동 결제하지 않음
- 로그아웃, 기기 변경이나 브라우저 저장소 삭제로 다시 받을 수 없음

## 2. 기능별 차감량

서버의 중앙 설정 파일 `plan-policy.mjs`는 `PLAN_CONFIG`, `AI_CREDIT_COSTS`, `AI_ACTION_LABELS`, `CREDIT_POLICY_VERSION`, `DEFAULT_TIME_ZONE`을 내보내고 다음 action과 차감량을 유일한 기준으로 사용합니다. 사용량 판정과 정규화는 `ai-credits-service.mjs`에서 이 값을 가져와야 하며 숫자를 다시 선언하지 않습니다.

```js
const AI_CREDIT_COSTS = {
  companion_chat: 1,
  create_daily_step: 2,
  revise_plan: 2,
  recovery_plan: 3,
  create_plan: 4,
  reschedule_plan: 4,
};
```

| action | 화면 표시 | 차감 |
| --- | --- | ---: |
| `companion_chat` | 올리와 지금 대화 | 1 크레딧 |
| `create_daily_step` | 오늘의 한 걸음 생성 | 2 크레딧 |
| `revise_plan` | 계획 일부 수정 | 2 크레딧 |
| `recovery_plan` | 회복 계획 생성 | 3 크레딧 |
| `create_plan` | 새 목표 계획 생성 | 4 크레딧 |
| `reschedule_plan` | 전체 일정 재조정 | 4 크레딧 |

프런트엔드는 예상 차감량을 설명할 수 있지만 최종 plan, action과 차감량을 결정할 수 없습니다. 서버는 클라이언트가 보낸 `plan`, 가격 또는 `creditCost`를 무시하고 인증된 사용자와 실제 라우트에 매핑된 action으로 직접 계산합니다.

## 3. 기간과 초기화

운영 결제가 비활성화된 현재 단계에서는 달력 기준으로 단순하고 예측 가능하게 운영합니다.

- 기준 시간대: 기존 사용자의 유효한 IANA `timezone`, 없거나 잘못되었으면 `Asia/Seoul`
- 일일 사용량 초기화: 사용자 시간대의 매일 00:00
- 월간 크레딧 초기화: 사용자 시간대의 매월 1일 00:00
- 체험 종료: 달력 경계가 아니라 `trialStartedAt + 24시간`
- 미사용 Free·Pro 월간 크레딧: 다음 기간으로 이월하지 않음

서버 로컬 시간이나 UTC 날짜 문자열을 그대로 기간 키로 사용하지 않습니다. 운영 결제가 활성화되면 Pro 월간 기간을 실제 결제 주기에 맞출 수 있도록 기간 계산과 결제 상태 판정을 분리합니다. 그 전까지는 결제 주기 기준으로 동작한다고 표시하지 않습니다.

## 4. 사용량 API 계약

인증된 사용자의 사용량 조회 응답은 서버가 계산한 현재 상태를 반환해야 합니다. 다음은 목표 계약이며, 배포 여부는 라우트 테스트로 확인합니다.

```json
{
  "ok": true,
  "plan": "free",
  "trial": {
    "eligible": true,
    "active": false,
    "endsAt": null,
    "remainingCredits": 0
  },
  "daily": {
    "used": 1,
    "limit": 2,
    "remaining": 1,
    "resetsAt": "ISO-8601"
  },
  "monthly": {
    "used": 3,
    "limit": 5,
    "remaining": 2,
    "resetsAt": "ISO-8601"
  },
  "creditCosts": {
    "companion_chat": 1,
    "create_daily_step": 2,
    "revise_plan": 2,
    "recovery_plan": 3,
    "create_plan": 4,
    "reschedule_plan": 4
  }
}
```

AI 성공 응답은 실제 차감량과 최신 잔여량을 함께 반환해야 합니다. 화면은 자체 계산보다 이 서버 응답을 우선합니다.

## 5. 예약, 확정과 실패 복구

AI 요청은 다음 상태 전이를 따릅니다.

1. 인증된 사용자와 서버 action을 결정한다.
2. 일일·월간 또는 체험 잔액을 검사한다.
3. 클라이언트가 생성한 고유 `requestId`로 필요한 크레딧을 예약한다.
4. 같은 사용자와 `requestId`가 이미 완료 또는 진행 중이면 다시 차감하지 않는다.
5. AI 공급자 호출이 유효한 결과로 끝나면 예약을 확정 차감한다.
6. 모델 오류, timeout, 네트워크 오류, 잘못된 응답 구조 또는 서버 오류이면 예약을 해제한다.
7. 해제된 요청은 확정 사용량에 포함하지 않는다.

공급자 호출 전에 잔액을 단순 차감한 뒤 클라이언트가 환불하도록 맡기지 않습니다. 클라이언트 연결이 끊겨도 서버가 예약 상태를 정리할 수 있어야 합니다. 10분 넘게 남은 예약은 다음 사용량 조회나 차감 판정 때 안전하게 해제합니다.

같은 `requestId`가 진행 중이면 `AI_REQUEST_IN_PROGRESS`, 이미 확정됐으면 `AI_REQUEST_ALREADY_COMMITTED`, 실패로 해제됐으면 `AI_REQUEST_PREVIOUSLY_RELEASED`를 HTTP 409로 반환하고 공급자를 다시 호출하지 않습니다. 원래 AI 결과 전문은 원장에 저장하거나 재전송하지 않으므로 이 409를 새 AI 성공 응답으로 표시해서는 안 됩니다. 자동 해제된 예약 뒤 늦게 도착한 공급자 결과도 `AI_CREDIT_RESERVATION_RELEASED`로 거부하여 재차감하지 않습니다.

## 6. KV schema-on-read 마이그레이션

별도 일괄 스크립트로 모든 사용자를 한 번에 수정하지 않고, `ai-credits-service.mjs`가 사용자를 읽을 때 새 정책 스키마로 정규화한 뒤 필요한 경우 다시 저장합니다. 별도 KV key나 인프라 migration을 만들지 않고 기존 `user:<id>` JSON 안의 `aiCredits` 필드를 사용합니다. 같은 사용자를 여러 번 읽어도 지급량이 늘지 않도록 정책 버전과 기간 키를 기록합니다.

전환 규칙은 다음과 같습니다.

1. 기존 유효 Pro 사용자는 Pro를 유지하고 새 서버 월간 기간을 시작합니다.
2. 기존 체험이 아직 유효하면 원래 종료 시각까지 인정합니다. 종료 후에는 활성 구독이 없으면 Free가 됩니다.
3. 만료되었거나 불완전한 체험 상태는 Free로 정규화합니다.
4. 그 밖의 기존 사용자는 Free로 시작합니다.
5. 브라우저의 레거시 `omwOllieEnergy` 값은 서버 크레딧으로 환산하지 않습니다.
6. 전환 시점에 새 서버 기간을 시작하고 해당 plan의 정규 할당량을 한 번만 제공합니다.
7. 목표·계획·기록·계정·구독 데이터는 보존합니다.
8. 이전 필드는 안정화 기간 동안 삭제하지 않아 롤백할 수 있게 합니다.

`aiCredits`의 논리 구조는 다음과 같습니다. 세부 하위 필드는 저장소 구현을 따르되 같은 의미를 다른 key에 중복 저장하지 않습니다.

- `policyVersion`, `timeZone`
- 체험 자격·시작·종료·사용량을 담는 `trial`
- 일일·월간 기간 키와 확정 사용량을 담는 `usage.day`, `usage.month`
- 진행 중 예약과 완료된 `requestId` 상태를 담는 `requests`
- action별 사용, token과 호출 수를 담는 `metrics.byAction`, `metrics.tokens`, `metrics.calls`

현재 plan과 구독 상태는 기존 사용자 레코드의 권위 있는 필드를 함께 판정하되 `aiCredits` 안에 서로 다른 진실을 중복 생성하지 않습니다. `payment_failed` 구독과 결제 기간이 지난 `active`·`canceled` 구독은 AI 요청 시점에도 Free로 판정하고, 아직 결제 기간이 남은 해지 예약은 종료 시각까지 Pro로 판정합니다. OAuth 프로필 갱신과 AI 원장 쓰기는 같은 isolate의 사용자 lock을 공유해 전체 사용자 레코드의 오래된 덮어쓰기를 막습니다. 체험 시작 시에는 결정론적 내부 사용자 ID에 연결된 가명 표식(`ai-trial-used:<userId>`)도 남겨, 계정 삭제 후 같은 Provider로 즉시 재가입해 체험을 다시 받는 경로를 막습니다. 탈퇴 직전에도 기존 체험 사용 시각으로 이 표식을 보강합니다. 이 표식은 일반 회원 데이터와 분리되고 체험 사용 후 1년이 지나면 무효화하므로, 탈퇴 후 방어 기간은 영구가 아니라 1년입니다.

## 7. KV 동시성의 잔여 위험

같은 Worker isolate에서는 사용자별 in-process lock으로 크레딧 작업을 직렬화하고, `requestId` 멱등성으로 같은 요청의 재전송을 막습니다. 하지만 Cloudflare KV의 일반적인 read-modify-write는 트랜잭션이 아니며 전역적으로 즉시 일관적이지 않습니다. 따라서 서로 다른 isolate·리전의 두 AI 요청이 동시에 같은 잔액을 읽는 경쟁까지 완전히 제거되지는 않으며, 드물게 한도를 초과해 승인되거나 한쪽 사용량 갱신이 유실될 수 있습니다.

따라서 다음을 운영 잔여 위험으로 취급합니다.

- UI의 버튼 잠금은 보조 수단일 뿐 보안 경계가 아님
- KV 기반 예약만으로 강한 원자성을 보장한다고 표현하지 않음
- 동시 요청, 재시도와 여러 기기 사용을 부하 테스트함
- 이상 사용량과 음수 잔액을 모니터링하고 보수적으로 차단함
- 결제 활성화나 사용량 증가 전 사용자별 Durable Object 또는 D1 트랜잭션 원장으로 이동하는 것을 우선 검토함

## 8. 결제 상태

- 정책상 Pro 가격은 월 4,900원입니다.
- Toss Payments 빌링 연동은 부분 구현 상태이며 운영은 `PAYMENTS_ENABLED=false`입니다.
- 결제 계약, 상품·MID 설정, 사업자·환불 고지, 테스트 결제와 영수증 검증을 마치기 전에는 실제 결제가 가능한 것처럼 표시하지 않습니다.
- 운영이 비활성화된 동안 CTA는 `Pro 출시 준비 중`으로 실제 상태를 반영하며, 전송 경로가 없는 출시 알림을 약속하지 않습니다.
- 추가 AI 크레딧 판매는 이번 범위에서 구현하지 않습니다.

현재 클라이언트 데이터 구조는 목표·활성 계획을 실제 다중 컬렉션으로 관리하지 않아 Pro의 여러 목표·계획 UX까지 제공하지는 않습니다. 중앙 정책의 Free 1개/Pro 제한 없음 값과 Free의 추가 생성 차단은 적용했지만, 다중 목표·계획 저장 모델과 화면은 별도 제품 작업이 필요합니다. 또한 `create_daily_step`의 비용과 안내 문구는 중앙 정책에 포함됐지만 독립 실행 API·버튼은 현재 저장소에 존재하지 않아 새 기능을 임의로 만들지 않았습니다.

완료된·실패한 `requestId` 기록은 현재 사용자 레코드에 남으며 별도 pruning 주기가 없습니다. 결제 활성화 또는 장기 운영 전 감사·재시도 기간을 정하고 보존·정리 정책을 구현해야 합니다.

## 9. 롤백

배포 후 사용량 또는 마이그레이션 이상이 발견되면 다음 순서로 롤백합니다.

1. AI 쓰기 경로를 중단하거나 안전 모드로 전환해 추가 차감을 막습니다.
2. 운영 Worker를 직전 정상 버전으로 되돌립니다.
3. schema-on-read가 남긴 이전 plan·trial·구독 필드를 삭제하지 않고 구버전이 읽도록 합니다.
4. 새 크레딧 상태와 예약 원장은 보존해 감사 자료로 사용하되 구버전 에너지로 역변환하지 않습니다.
5. 진행 중 예약을 만료·해제하고 확정 호출과 대조합니다.
6. 기존 Pro를 Free로 강등하지 않고, 유효 체험 종료 시각과 목표·계획 데이터를 보존합니다.
7. 수정 버전 배포 시 정책 버전을 올리고 같은 사용자에게 크레딧이 다시 지급되지 않는지 검증합니다.

KV 데이터는 즉시 대량 삭제하지 않습니다. 되돌리기 전에 사용자 수, plan 분포, 기간별 사용량과 미해제 예약을 내보내 비교합니다.

## 10. 출시 검증
